Annotation

  • System routes patient to the consultation form for that product category (e.g. Hair Loss)
  • The product and category are passed through so the correct form loads
  • If not logged in, they create an account or log in first
  • Account creation collects: full name, DOB, email, phone, delivery address

Annotation

Payment model

  • Payment is taken upfront, in full — not a pre-auth hold
  • If the order is later rejected (ID fail or clinical denial), a full refund is issued
  • Auto-denial (Phase 2) happens before payment — no refund needed in that case

Record saved

  • Patient ID, all Q&As (mapped to questions), timestamp, product/category, unique reference number
  • Both POM and P items go through ID check next — review role differs later

Notification email (secure, limited details)

  • Contains: consultation reference, product category, type (POM/P), date/time, direct link to case

  • Patient rejection email sent only after refund is confirmed

Annotation

Where things happen

  • Vivamen backend: treatment review, patient notes, clinical decision
  • SignatureRx: the prescription itself is created, written, and signed

API flow (two calls — one out, one back)

  • Website → SignatureRx: CREATE PRESCRIPTION — sends all Rx fields, receives reference ID
  • SignatureRx → Website: WEBHOOK — sends signed Rx details, SHA-256 hash, timestamp, prescriber details
  • Everything between these calls (MFA, signing) is handled entirely within SignatureRx

Rx fields (auto-populated)

Patient name, DOB, medication name, strength, form, dose, frequency, quantity/duration, directions, date of issue, prescriber identity. Prescriber can adjust any field except name and DOB.

Version history (visible in backend)
